プラガビリティ

エラーレベルとオプション

念のため、エラーレベルとそのロギングはdevelopment.yamlで制御できます。

ロガー

# Application logging configuration
logger:
  # Enable or disable logging.
  enable: true
  # Enable pretty backtrace (sets RUST_BACKTRACE=1)
  pretty_backtrace: true
  # Log level, options: trace, debug, info, warn or error.
  level: debug
  # Define the logging format. options: compact, pretty or json
  format: compact
  # By default the logger has filtering only logs that came from your code or logs that came from `loco` framework. to see all third party libraries
  # Uncomment the line below to override to see all third party libraries you can enable this config and override the logger filters.
  # override_filter: trace

ここで最も重要な設定は以下の通りです。

  • level - 標準のロギングレベルです。通常、開発環境ではdebugまたはtraceです。本番環境では、慣れているものを選んでください。
  • pretty_backtrace - エラーの原因となったコード行への明確で簡潔なパスを提供します。開発環境ではtrueを使い、本番環境ではオフにしてください。本番環境でデバッグが必要な場合は、オンにして、完了したらオフにすることができます。

コントローラーのロギング

server.middlewaresには以下があります。

server:
  middlewares:
    #
    # ...
    #
    # Generating a unique request ID and enhancing logging with additional information such as the start and completion of request processing, latency, status code, and other request details.
    logger:
      # Enable/Disable the middleware.
      enable: true

これを有効にすると、詳細なリクエストエラーと、複数のリクエストスコープのエラーを照合するのに役立つ便利なrequest-idを取得できます。

データベース

databaseセクションで、ライブSQLクエリをログに記録するオプションがあります。

database:
  # When enabled, the sql query will be logged.
  enable_logging: false

エラーの処理

アプリケーション開発中は、主にターミナルでエラーを確認することになります。以下のような表示になります。

2024-02-xxx DEBUG http-request: tower_http::trace::on_request: started processing request http.method=GET http.uri=/notes http.version=HTTP/1.1 http.user_agent=curl/8.1.2 environment=development request_id=8622e624-9bda-49ce-9730-876f2a8a9a46
2024-02-xxx11T12:19:25.295954Z ERROR http-request: loco_rs::controller: controller_error error.msg=invalid type: string "foo", expected a sequence error.details=JSON(Error("invalid type: string \"foo\", expected a sequence", line: 0, column: 0)) error.chain="" http.method=GET http.uri=/notes http.version=HTTP/1.1 http.user_agent=curl/8.1.2 environment=development request_id=8622e624-9bda-49ce-9730-876f2a8a9a46

通常、エラーからは以下の情報が期待できます。

  • error.msg - オペレーター向けの、エラーのto_string()バージョンです。
  • error.detail - 開発者向けの、エラーのデバッグ表現です。
  • 口頭でのエラーメッセージではなく、検索用に調整された主要メッセージとして、controller_errorのようなエラータイプが表示されます。
  • エラーは*トレース*イベントおよびスパンとしてログに記録されるため、カスタムのトレースサブスクライバーを提供する任意のインフラストラクチャを構築できます。loco-extrasのprometheusの例を参照してください。

注意事項:

  • *エラーチェーン*は実験されましたが、実際にはほとんど価値がありません。
  • エンドユーザーが目にするエラーは全く別のものです。ユーザーがエラーに対して何もできない場合（例：「データベースオフラインエラー」など）、セキュリティ上の理由から、エンドユーザーにはエラーに関する最小限の内部詳細のみを提供するよう努めており、ほとんどの場合、意図的に一般的な「内部サーバーエラー」となります。

エラーの生成

コントローラーを構築する際、ハンドラーはResult<impl IntoResponse>を返すように記述します。ここでのResultはLocoのResultであり、LocoのError型も関連付けられていることを意味します。

LocoのError型を使用する場合、以下のいずれかをレスポンスとして使用できます。

Err(Error::string("some custom message"));
Err(Error::msg(other_error)); // turns other_error to its string representation
Err(Error::wrap(other_error));
Err(Error::Unauthorized("some message"))

// or through controller helpers:
unauthorized("some message") // create a full response object, calling Err on a created error

初期化子

初期化子は、アプリケーションで行う必要のあるインフラストラクチャの「配線」をカプセル化する方法です。初期化子はsrc/initializers/に配置します。

初期化子の記述

現在、初期化子とはInitializerトレイトを実装するものです。

pub trait Initializer: Sync + Send {
    /// The initializer name or identifier
    fn name(&self) -> String;

    /// Occurs after the app's `before_run`.
    /// Use this to for one-time initializations, load caches, perform web
    /// hooks, etc.
    async fn before_run(&self, _app_context: &AppContext) -> Result<()> {
        Ok(())
    }

    /// Occurs after the app's `after_routes`.
    /// Use this to compose additional functionality and wire it into an Axum
    /// Router
    async fn after_routes(&self, router: AxumRouter, _ctx: &AppContext) -> Result<AxumRouter> {
        Ok(router)
    }
}

例：Axumセッションの統合

axum-sessionを使用してアプリケーションにセッションを追加したいと考えるかもしれません。また、その機能を自分のプロジェクト間で共有したり、他の人からそのコードを入手したりしたいと考えるかもしれません。

統合を*初期化子*としてコーディングすれば、この再利用を簡単に実現できます。

// place this in `src/initializers/axum_session.rs`
#[async_trait]
impl Initializer for AxumSessionInitializer {
    fn name(&self) -> String {
        "axum-session".to_string()
    }

    async fn after_routes(&self, router: AxumRouter, _ctx: &AppContext) -> Result<AxumRouter> {
        let session_config =
            axum_session::SessionConfig::default().with_table_name("sessions_table");
        let session_store =
            axum_session::SessionStore::<axum_session::SessionNullPool>::new(None, session_config)
                .await
                .unwrap();
        let router = router.layer(axum_session::SessionLayer::new(session_store));
        Ok(router)
    }
}

そして、アプリケーションの構造は以下のようになります。

src/
 bin/
 controllers/
    :
    :
 initializers/       <--- 新しいフォルダ
   mod.rs            <--- 新しいモジュール
   axum_session.rs   <--- あなたの新しい初期化子
    :
    :
  app.rs   <--- ここに初期化子を登録

初期化子の使用

独自の初期化子を実装したら、src/app.rsにinitializers(..)フックを実装し、初期化子のVecを提供する必要があります。

    async fn initializers(_ctx: &AppContext) -> Result<Vec<Box<dyn Initializer>>> {
        let initializers: Vec<Box<dyn Initializer>> = vec![
            Box::new(initializers::axum_session::AxumSessionInitializer),
            Box::new(initializers::view_engine::ViewEngineInitializer),
            Box::new(initializers::hello_view_engine::HelloViewEngineInitializer),
        ];

        Ok(initializers)
    }

Locoは、アプリの起動プロセス中に、正しい場所で初期化子スタックを実行します。

他にできることは？

現在、初期化子には2つの統合ポイントがあります。

  • before_run - アプリ実行前に発生します。これは純粋な「初期化」タイプのフックです。ウェブフックの送信、メトリックポイントの送信、クリーンアップ、事前チェックなどを行うことができます。
  • after_routes - ルートが追加された後に発生します。Axumルーターとその強力なレイヤー統合ポイントにアクセスでき、ここでほとんどの時間を費やすことになります。

Railsの初期化子との比較

Railsの初期化子は、初期化のために一度だけ実行され、すべてにアクセスできる通常のスクリプトです。これらは「ライブ」なRailsアプリケーションにアクセスし、グローバルインスタンスとして変更できることからその力を得ています。

Locoでは、Rustではグローバルインスタンスにアクセスして変更することはできません（正当な理由があります！）。そのため、明示的で安全な2つの統合ポイントを提供しています。

  1. 純粋な初期化（設定されたアプリへの影響なし）
  2. 実行中のアプリとの統合（Axumルーター経由）

Railsの初期化子には*順序付け*と*変更*が必要です。つまり、ユーザーは特定の順序で実行されることを確信し（または再順序付けし）、他の人が事前に設定した初期化子を削除できる必要があります。

Locoでは、ユーザーに初期化子の*完全なVecを提供する*ことで、この複雑さを回避しています。Vecは順序付けされており、暗黙の初期化子はありません。

グローバルロガー初期化子

一部の開発者は、ロギングスタックをカスタマイズしたいと考えるかもしれません。Locoでは、これにはトレースとトレースサブスクライバーの設定が含まれます。

現在のところ、トレースでは再初期化や実行中のトレーススタックの変更が許可されていないため、グローバルなトレーススタックを初期化して登録する機会は*一度しかありません*。

これが、独自のロギングスタック初期化を提供するために使用できる、init_loggerという新しい*アプリレベルのフック*を追加した理由です。

// in src/app.rs
impl Hooks for App {
    // return `Ok(true)` if you took over initializing logger
    // otherwise, return `Ok(false)` to use the Loco logging stack.
    fn init_logger(_config: &config::Config, _env: &Environment) -> Result<bool> {
        Ok(false)
    }
}

独自のロガーを設定したら、初期化を引き継いだことを示すためにOk(true)を返します。

ミドルウェア

Locoは、axumとtowerの上に構築されたフレームワークです。これらは、layerとserviceをミドルウェアとしてルートやハンドラーに追加する方法を提供します。

ミドルウェアは、リクエストに前処理と後処理を追加する方法です。これは、ロギング、認証、レート制限、ルート固有の処理などに使用できます。

ソースコード

Locoのルートミドルウェア/レイヤーの実装は、axumのRouter::layerに似ています。ミドルウェアのソースコードは、src/controllers/routesディレクトリで見つけることができます。このlayer関数は、ミドルウェアレイヤーをルートの各ハンドラーにアタッチします。

// src/controller/routes.rs
use axum::{extract::Request, response::IntoResponse, routing::Route};
use tower::{Layer, Service};

impl Routes {
    pub fn layer<L>(self, layer: L) -> Self
        where
            L: Layer<Route> + Clone + Send + 'static,
            L::Service: Service<Request> + Clone + Send + 'static,
            <L::Service as Service<Request>>::Response: IntoResponse + 'static,
            <L::Service as Service<Request>>::Error: Into<Infallible> + 'static,
            <L::Service as Service<Request>>::Future: Send + 'static,
    {
        Self {
            prefix: self.prefix,
            handlers: self
                .handlers
                .iter()
                .map(|handler| Handler {
                    uri: handler.uri.clone(),
                    actions: handler.actions.clone(),
                    method: handler.method.clone().layer(layer.clone()),
                })
                .collect(),
        }
    }
}

基本的なミドルウェア

この例では、リクエストメソッドとパスをログに記録する基本的なミドルウェアを作成します。

// src/controllers/middleware/log.rs
use std::{
    convert::Infallible,
    task::{Context, Poll},
};

use axum::{
    body::Body,
    extract::{FromRequestParts, Request},
    response::Response,
};
use futures_util::future::BoxFuture;
use loco_rs::prelude::{auth::JWTWithUser, *};
use tower::{Layer, Service};

use crate::models::{users};

#[derive(Clone)]
pub struct LogLayer;

impl LogLayer {
    pub fn new() -> Self {
        Self {}
    }
}

impl<S> Layer<S> for LogLayer {
    type Service = LogService<S>;

    fn layer(&self, inner: S) -> Self::Service {
        Self::Service {
            inner,
        }
    }
}

#[derive(Clone)]
pub struct LogService<S> {
    // S is the inner service, in the case, it is the `/auth/register` handler
    inner: S,
}

/// Implement the Service trait for LogService
/// # Generics
/// * `S` - The inner service, in this case is the `/auth/register` handler
/// * `B` - The body type
impl<S, B> Service<Request<B>> for LogService<S>
    where
        S: Service<Request<B>, Response=Response<Body>, Error=Infallible> + Clone + Send + 'static, /* Inner Service must return Response<Body> and never error, which is typical for handlers */
        S::Future: Send + 'static,
        B: Send + 'static,
{
    // Response type is the same as the inner service / handler
    type Response = S::Response;
    // Error type is the same as the inner service / handler
    type Error = S::Error;
    // Future type is the same as the inner service / handler
    type Future = BoxFuture<'static, Result<Self::Response, Self::Error>>;

    // poll_ready is used to check if the service is ready to process a request
    fn poll_ready(&mut self, cx: &mut Context<'_>) -> Poll<Result<(), Self::Error>> {
        // Our middleware doesn't care about backpressure, so it's ready as long
        // as the inner service is ready.
        self.inner.poll_ready(cx)
    }

    fn call(&mut self, req: Request<B>) -> Self::Future {
        let clone = self.inner.clone();
        // take the service that was ready
        let mut inner = std::mem::replace(&mut self.inner, clone);
        Box::pin(async move {
            let (mut parts, body) = req.into_parts();
            tracing::info!("Request: {:?} {:?}", parts.method, parts.uri.path());
            let req = Request::from_parts(parts, body);
            inner.call(req).await
        })
    }
}

一見すると、このミドルウェアは少し圧倒されます。分解してみましょう。

LogLayerは、内部サービスをラップするtower::Layerです。

LogServiceは、リクエスト用のServiceトレイトを実装するtower::Serviceです。

ジェネリクス（総称型）の説明

Layer

Layerトレイトでは、Sは内部サービスを表し、この場合は/auth/registerハンドラーです。layer関数は、この内部サービスを受け取り、それをラップする新しいサービスを返します。

Service

Sは内部サービスであり、この場合は/auth/registerハンドラーです。ハンドラーに使用するget、post、put、delete関数を見ると、それらはすべてMethodRoute<S, Infallible>（これはサービスです）を返します。

したがって、S: Service<Request<B>, Response = Response<Body>, Error = Infallible>は、Request<B>（ボディを持つリクエスト）を受け取り、Response<Body>を返すことを意味します。ErrorがInfallibleであるということは、ハンドラーがエラーを発生させないことを意味します。

S::Future: Send + 'staticは、内部サービスのFutureはSendトレイトと'staticを実装している必要があります。

type Response = S::Responseは、ミドルウェアのレスポンス型は内部サービスと同じです。

type Error = S::Errorは、ミドルウェアのエラー型は内部サービスと同じです。

type Future = BoxFuture<'static, Result<Self::Response, Self::Error>>は、ミドルウェアのFuture型は内部サービスと同じです。

B: Send + 'staticは、リクエストボディ型はSendトレイトと'staticを実装している必要があります。

関数説明

LogLayer

LogLayer::new関数は、LogLayerの新しいインスタンスを作成するために使用されます。

LogService

LogService::poll_ready関数は、サービスがリクエストを処理する準備ができているかどうかを確認するために使用されます。これはバックプレッシャーに使用できます。詳細については、tower::ServiceのドキュメントとTokioのチュートリアルを参照してください。

LogService::call関数は、リクエストを処理するために使用されます。この場合、リクエストメソッドとパスをログに記録しています。その後、リクエストとともに内部サービスを呼び出しています。

poll_readyの重要性:

Towerフレームワークでは、サービスがリクエストを処理するために使用される前に、poll_readyメソッドを使用して準備ができているかを確認する必要があります。このメソッドは、サービスがリクエストを処理する準備ができている場合にPoll::Ready(Ok(()))を返します。サービスが準備できていない場合、Poll::Pendingを返し、呼び出し元がリクエストを送信する前に待機すべきであることを示します。このメカニズムは、サービスがリクエストを効率的かつ正確に処理するために必要なリソースまたは状態を持っていることを保証します。

クローンと準備状態

サービスをクローンする際、特にボックス化されたFutureまたは同様のコンテキストに移動させる場合、クローンが元のサービスの準備状態を継承しないことを理解することが重要です。サービスの各クローンは独自の状態を維持します。これは、元のサービスが準備完了状態(Poll::Ready(Ok(())))であっても、クローンされたサービスがクローン直後に同じ状態でない可能性があることを意味します。これにより、クローンされたサービスが準備完了する前に使用され、パニックやその他の障害を引き起こす可能性があります。

std::mem::replaceを使用したサービスのクローニングの正しいアプローチ
クローニングを正しく処理するには、準備ができたサービスをそのクローンと制御された方法で交換するためにstd::mem::replaceを使用することをお勧めします。このアプローチにより、リクエストを処理するために使用されているサービスが、準備ができたと検証されたものであることが保証されます。動作は以下のとおりです。

  • サービスをクローンする: まず、サービスのクローンを作成します。このクローンは、最終的にサービスハンドラーの元のサービスを置き換えます。
  • 元のサービスをクローンと置き換える: std::mem::replaceを使用して、元のサービスをクローンと交換します。この操作により、サービスハンドラーがサービスインスタンスを保持し続けることが保証されます。
  • 元のサービスを使用してリクエストを処理する: 元のサービスは既に準備状態が確認されているため（poll_ready経由）、それを使用して受信リクエストを処理することは安全です。現在ハンドラーにあるクローンは、次回準備状態が確認されるものになります。

このメソッドは、リクエストを処理するために使用される各サービスインスタンスが、常に明示的に準備状態が確認されたものであることを保証し、それによってサービス処理の整合性と信頼性を維持します。

このパターンを示す簡略化された例を次に示します。

// Wrong
fn call(&mut self, req: Request<B>) -> Self::Future {
    let mut inner = self.inner.clone();
    Box::pin(async move {
        /* ... */
        inner.call(req).await
    })
}

// Correct
fn call(&mut self, req: Request<B>) -> Self::Future {
    let clone = self.inner.clone();
    // take the service that was ready
    let mut inner = std::mem::replace(&mut self.inner, clone);
    Box::pin(async move {
        /* ... */
        inner.call(req).await
    })
}

この例では、innerは準備ができていたサービスであり、リクエストを処理した後、self.innerはクローンを保持し、これは次のサイクルで準備状態がチェックされます。サービス準備状態とクローニングのこの慎重な管理は、Towerを使用した非同期Rustアプリケーションで堅牢でエラーのないサービス操作を維持するために不可欠です。

Tower Serviceクローニングドキュメント

ハンドラーへのミドルウェア追加

auth::registerハンドラーにミドルウェアを追加します。

// src/controllers/auth.rs
pub fn routes() -> Routes {
    Routes::new()
        .prefix("auth")
        .add("/register", post(register).layer(middlewares::log::LogLayer::new()))
}

これで、auth::registerハンドラーにリクエストを送信すると、リクエストメソッドとパスがログに記録されるようになります。

2024-XX-XXTXX:XX:XX.XXXXXZ  INFO http-request: xx::controllers::middleware::log Request: POST "/auth/register" http.method=POST http.uri=/auth/register http.version=HTTP/1.1  environment=development request_id=xxxxx

ルートへのミドルウェア追加

authルートにミドルウェアを追加します。

// src/main.rs
pub struct App;

#[async_trait]
impl Hooks for App {
    fn routes(_ctx: &AppContext) -> AppRoutes {
        AppRoutes::with_default_routes()
            .add_route(
                controllers::auth::routes()
                    .layer(middlewares::log::LogLayer::new()),
            )
    }
}

これで、authルート内の任意のハンドラーにリクエストを送信すると、リクエストメソッドとパスがログに記録されるようになります。

2024-XX-XXTXX:XX:XX.XXXXXZ  INFO http-request: xx::controllers::middleware::log Request: POST "/auth/register" http.method=POST http.uri=/auth/register http.version=HTTP/1.1  environment=development request_id=xxxxx

高度なミドルウェア (AppContext付き)

ミドルウェアでAppContextにアクセスする必要がある場合があります。例えば、データベース接続にアクセスして認可チェックを実行したい場合などです。これを行うには、AppContextをLayerとServiceに追加します。

ここでは、JWTトークンをチェックし、データベースからユーザーを取得してユーザー名を出力するミドルウェアを作成します。

// src/controllers/middleware/log.rs
use std::{
    convert::Infallible,
    task::{Context, Poll},
};

use axum::{
    body::Body,
    extract::{FromRequestParts, Request},
    response::Response,
};
use futures_util::future::BoxFuture;
use loco_rs::prelude::{auth::JWTWithUser, *};
use tower::{Layer, Service};

use crate::models::{users};

#[derive(Clone)]
pub struct LogLayer {
    state: AppContext,
}

impl LogLayer {
    pub fn new(state: AppContext) -> Self {
        Self { state }
    }
}

impl<S> Layer<S> for LogLayer {
    type Service = LogService<S>;

    fn layer(&self, inner: S) -> Self::Service {
        Self::Service {
            inner,
            state: self.state.clone(),
        }
    }
}
