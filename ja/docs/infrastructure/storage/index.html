ストレージ

Locoストレージでは、複数の操作を通じてファイルを簡単に扱えるようにします。ストレージは、インメモリ、ディスク上、またはAWS S3、GCP、Azureなどのクラウドサービスを利用できます。

Locoは、シンプルなストレージ操作に加え、データのミラーリングや異なる障害モードを持つバックアップ戦略などの高度な機能をサポートしています。

デフォルトでは、インメモリおよびディスクストレージがすぐに利用できます。クラウドプロバイダーと連携するには、以下の機能を指定する必要があります。

  • storage_aws_s3
  • storage_azure
  • storage_gcp
  • all_storage

デフォルトでは、LocoはNullプロバイダーを初期化します。これは、ストレージに対するいかなる操作もエラーを返すことを意味します。

セットアップ

app.rsファイルにafter_context関数をフックとして追加し、loco_rsからstorageモジュールをインポートします。

use loco_rs::storage;

async fn after_context(ctx: AppContext) -> Result<AppContext> {
    Ok(ctx)
}

このフックは、次のセクションで説明するすべてのストレージ設定を保持するストレージインスタンスを返します。このストレージインスタンスは、アプリケーションコンテキストの一部として保存され、コントローラ、エンドポイント、タスクワーカーなどで利用できます。

用語集
StorageDriverストレージを行うトレイト実装
Storage1つ以上のストレージドライバーを管理するための抽象化実装。
Strategyミラーやバックアップなど、ストレージの様々な戦略を実装するトレイト。
FailureMode各戦略内で実装され、障害発生時の操作の処理方法を決定します。

ストレージの初期化

ストレージは、単一のドライバーまたは複数のドライバーで設定できます。

単一ストレージ

この例では、インメモリドライバーを初期化し、single関数で新しいストレージを作成します。

use loco_rs::storage;

async fn after_context(ctx: AppContext) -> Result<AppContext> {
    Ok(AppContext {
        storage: Storage::single(storage::drivers::mem::new()).into(),
        ..ctx
    })
}

複数のドライバー

高度な使用法として、複数のドライバーを設定し、すぐに利用できるスマートな戦略を適用できます。各戦略には、独自の一連の障害モードがあり、それらをどのように処理するかを決定できます。

複数のドライバーを作成する:

use crate::storage::{drivers, Storage};

let aws_1 = drivers::aws::new("users");
let azure = drivers::azure::new("users");
let aws_2 = drivers::aws::new("users-mirror");

ミラー戦略:

ミラーサービスを定義することで、複数のサービスを同期状態に保つことができます。ミラーサービスは、2つ以上の下位サービス間でアップロード、削除、名前変更、コピーを**複製**します。ダウンロードの動作は冗長的にデータを取得します。つまり、プライマリからのファイル取得が失敗した場合、セカンダリで最初に見つかったファイルが返されます。

動作

3つのストアインスタンスを作成した後、ミラー戦略インスタンスを作成し、障害モードを定義する必要があります。ミラー戦略は、プライマリストアとセカンダリストアのリスト、および障害モードオプションを期待します。

  • MirrorAll: すべてのセカンダリストレージが成功する必要があります。いずれかが失敗した場合でも、操作は残りのストレージに対して続行されますが、エラーを返します。
  • AllowMirrorFailure: 1つ以上のミラー操作が失敗した場合でも、操作はエラーを返しません。

障害モードは、アップロード、削除、移動、コピーに関連します。

例:


// Define the mirror strategy by setting the primary store and secondary stores by names.
let strategy = Box::new(MirrorStrategy::new(
    "store_1",
    Some(vec!["store_2".to_string(), "store_3".to_string()]),
    FailureMode::MirrorAll,
)) as Box<dyn StorageStrategy>;

// Create the storage with the store mapping and the strategy.
 let storage = Storage::new(
    BTreeMap::from([
        ("store_1".to_string(), aws_1),
        ("store_2".to_string(), azure),
        ("store_3".to_string(), aws_2),
    ]),
    strategy.into(),
);

バックアップ戦略:

複数のストレージにわたって操作をバックアップし、障害モードポリシーを制御できます。

3つのストアインスタンスを作成した後、バックアップ戦略インスタンスを作成し、障害モードを定義する必要があります。バックアップ戦略は、プライマリストアとセカンダリストアのリスト、および障害モードオプションを期待します。

  • BackupAll: すべてのセカンダリストレージが成功する必要があります。いずれかが失敗した場合でも、操作は残りのストレージに対して続行されますが、エラーを返します。
  • AllowBackupFailure: 1つ以上のバックアップ操作が失敗した場合でも、操作はエラーを返しません。
  • AtLeastOneFailure: 少なくとも1つの操作は成功する必要があります。
  • CountFailure: 指定された数のバックアップが成功する必要があります。

障害モードは、アップロード、削除、移動、コピーに関連します。ダウンロードは常にプライマリからファイルを取得します。

例:


// Define the backup strategy by setting the primary store and secondary stores by names.
let strategy: Box<dyn StorageStrategy> = Box::new(BackupStrategy::new(
    "store_1",
    Some(vec!["store_2".to_string(), "store_3".to_string()]),
    FailureMode::AllowBackupFailure,
)) as Box<dyn StorageStrategy>;

let storage = Storage::new(
    BTreeMap::from([
        ("store_1".to_string(), store_1),
        ("store_2".to_string(), store_2),
        ("store_3".to_string(), store_3),
    ]),
    strategy.into(),
);

独自の戦略を作成する

特定の戦略がある場合は、StorageStrategyを実装し、すべてのストア機能を実装することで、簡単に独自の戦略を作成できます。

コントローラでの使用

この例に従い、axumクレートでmultipart機能を有効にしてください。

async fn upload_file(
    State(ctx): State<AppContext>,
    mut multipart: Multipart,
) -> Result<Response> {
    let mut file = None;
    while let Some(field) = multipart.next_field().await.map_err(|err| {
        tracing::error!(error = ?err,"could not readd multipart");
        Error::BadRequest("could not readd multipart".into())
    })? {
        let file_name = match field.file_name() {
            Some(file_name) => file_name.to_string(),
            _ => return Err(Error::BadRequest("file name not found".into())),
        };

        let content = field.bytes().await.map_err(|err| {
            tracing::error!(error = ?err,"could not readd bytes");
            Error::BadRequest("could not readd bytes".into())
        })?;

        let path = PathBuf::from("folder").join(file_name);
        ctx.storage.as_ref().upload(path.as_path(), &content).await?;

        file = Some(path);
    }

    file.map_or_else(not_found, |path| {
        format::json(views::upload::Response::new(path.as_path()))
    })
}

テスト

コントローラでファイルストレージをテストするには、この例に従うことができます。

use loco_rs::testing::prelude::*;

#[tokio::test]
#[serial]
async fn can_register() {
    request::<App, _, _>(|request, ctx| async move {
        let file_content = "loco file upload";
        let file_part = Part::bytes(file_content.as_bytes()).file_name("loco.txt");

        let multipart_form = MultipartForm::new().add_part("file", file_part);

        let response = request.post("/upload/file").multipart(multipart_form).await;

        response.assert_status_ok();

        let res: views::upload::Response = serde_json::from_str(&response.text()).unwrap();

        let stored_file: String = ctx.storage.as_ref().download(&res.path).await.unwrap();

        assert_eq!(stored_file, file_content);
    })
    .await;
}
