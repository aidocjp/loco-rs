モデル

`loco`のモデルは、簡単なデータベースクエリや書き込み、さらにマイグレーションやシーディングを可能にするエンティティクラスを意味します。

SqliteとPostgres

新しいアプリを作成した際にデフォルトで`sqlite`を選択したかもしれません。Locoでは、`sqlite`と`postgres`の間をシームレスに移行できます。

開発には`sqlite`を、本番には`postgres`を使用するのが一般的です。`pg`固有の機能を使用するため、開発と本番の両方で`postgres`を好む人もいます。最近では、本番でも`sqlite`を使用する人もいます。いずれにしても、すべて有効な選択肢です。

`sqlite`の代わりに`postgres`を設定するには、`config/development.yaml`（または`production.yaml`）を開き、アプリ名が`myapp`であると仮定して、以下のように設定します。

database:
  uri: "{{ get_env(name="DATABASE_URL", default="[REDACTED_DATABASE_URL] }"
ローカルのPostgresデータベースは`loco:loco`で、DB名は`myapp_development`とする必要があります。テスト環境と本番環境では、それぞれ`myapp_test`と`myapp_production`というDB名にする必要があります。

参考までに、PostgreSQLデータベースサーバーを起動するためのDockerコマンドを以下に示します。

docker run -d -p 5432:5432 \
  -e POSTGRES_USER=loco \
  -e POSTGRES_DB=myapp_development \
  -e POSTGRES_PASSWORD="loco" \
  postgres:15.3-alpine

最後に、doctorコマンドを使って接続を検証することもできます。

$ cargo loco doctor
    Finished dev [unoptimized + debuginfo] target(s) in 0.32s
    Running `target/debug/myapp-cli doctor`
✅ SeaORM CLI is installed
✅ DB connection: success
✅ Redis connection: success

ファットモデル、スリムコントローラ

`loco`のモデルは**アクティブレコードをベースに設計されています**。これは、モデルがアプリケーションの中心であり、すべてのロジックや操作がそこに存在するべきであることを意味します。

これは、`User::create`がユーザーを作成するだけでなく、`user.buy(product)`が製品を購入する、という意味です。

この方向性に同意すれば、以下のメリットを無料で得られます。

  • モデルのテストでほとんどすべてのロジックと可動部分がテストされるため、**時間効率の良いテスト**が可能です。
  • **タスクから、またはワーカーや他の場所から**、完全なアプリのワークフローを実行する機能。
  • モデルのみを組み合わせることで、**機能やユースケースを効果的に構成**できます。
  • 要するに、**モデルがあなたのアプリになり**、コントローラはアプリを世界に公開する単なる手段となります。

私たちのアクティブレコード抽象化の背後にある主要なORMとして、`SeaORM`を使用しています。

  • Dieselではないのはなぜ？ - Dieselはパフォーマンスは優れていますが、そのマクロと一般的なアプローチは、私たちがやろうとしていることと相容れないと感じました。
  • sqlxではないのはなぜ？ - SeaORMは内部でsqlxを使用しているため、必要であれば生の`sqlx`を使用するための基盤は整っています。

モデルの例

`loco`モデルのライフサイクルはマイグレーションから始まり、その後データベース構造からエンティティのRustコードが自動的に生成されます。

src/
  models/
    _entities/   <--- 自動生成コード
      users.rs   <--- 基本エンティティとヘルパートレイト
    users.rs  <--- カスタムアクティブレコードコード

ユーザーのアクティブレコードの使用方法は、SeaORMでの使用方法と同じです。例はこちらを参照

ユーザーアクティブレコードに機能を追加するには、拡張機能を使用します。

impl super::_entities::users::ActiveModel {
    /// .
    ///
    /// # Errors
    ///
    /// .
    pub fn foobar(&self) -> Result<(), DbErr> {
        // implement and get back a `user.foobar()`
    }
}

モデルの作成

モデルジェネレータ

新しいモデルを追加するには、モデルジェネレータがマイグレーションを作成し、それを実行し、次にデータベーススキーマからエンティティ同期をトリガーして、モデルエンティティをハイドレートし作成します。

$ cargo loco generate model posts title:string! content:text user:references

マイグレーションを介してモデルが追加されると、以下のデフォルトフィールドが提供されます。

  • `created_at` (ts!): モデルが作成されたことを示すタイムスタンプです。
  • `updated_at` (ts!): モデルが更新されたことを示すタイムスタンプです。

これらのフィールドは、マイグレーションコマンドで提供された場合、無視されます。

マイグレーション

モデルジェネレータを使用する以外に、マイグレーションを作成することでスキーマを操作します。

$ cargo loco generate migration <name of migration> [name:type, name:type ...]

これにより、プロジェクトのルートにある`migration/`にマイグレーションが作成されます。

適用できます。

$ cargo loco db migrate

そして、そこからエンティティ（Rustコード）を生成し直します。

LocoはRailsに似たマイグレーションファーストのフレームワークです。これは、モデル、データフィールド、またはモデル指向の変更を追加したい場合、それを記述するマイグレーションから始め、そのマイグレーションを適用することで`model/_entities`に生成されたエンティティを取り戻すことを意味します。

これにより、**すべてをコードとして表現**し、**再現性**と**原子性**が強制され、スキーマの知識が失われることはありません。

マイグレーションの名前付けは重要であり、生成されるマイグレーションの種類はマイグレーション名から推論されます。

新しいテーブルを作成する

  • 名前テンプレート: `Create___`
  • 例: `CreatePosts`
$ cargo loco g migration CreatePosts title:string content:string

カラムを追加する

  • 名前テンプレート: `Add___To___`
  • 例: `AddNameAndAgeToUsers`（文字列`NameAndAge`は重要ではありません。カラムは個別に指定しますが、`Users`はテーブル名になるため重要です。）
$ cargo loco g migration AddNameAndAgeToUsers name:string age:int

カラムを削除する

  • 名前テンプレート: `Remove___From___`
  • 例: `RemoveNameAndAgeFromUsers`（カラム追加の場合と同じ注記）
$ cargo logo g migration RemoveNameAndAgeFromUsers name:string age:int

参照を追加する

  • 名前テンプレート: `Add___RefTo___`
  • 例: `AddUserRefToPosts`（Userは重要ではありません。参照は個別に1つまたは複数指定するためです。Postsはマイグレーションのテーブル名になるため重要です。）
$ cargo loco g migration AddUserRefToPosts user:references

結合テーブルを作成する

  • 名前テンプレート: `CreateJoinTable___And___`（2つのテーブル間でサポート）
  • 例: `CreateJoinTableUsersAndGroups`
$ cargo loco g migration CreateJoinTableUsersAndGroups count:int

リレーションシップに関する状態カラム（ここでの`count`など）を追加することもできます。

空のマイグレーションを作成する

上記のパターンに当てはまらないマイグレーションには、記述的な名前を使用して空のマイグレーションを作成します。

$ cargo loco g migration FixUsersTable

ダウンマイグレーション

もし間違いに気づいたら、いつでもマイグレーションを元に戻すことができます。これにより、マイグレーションによって行われた変更が元に戻されます（マイグレーションに適切な`down`のコードを追加していることを前提とします）。

cargo loco db down

`down`コマンド単独では、最後のマイグレーションのみをロールバックします。複数のマイグレーションをロールバックしたい場合は、ロールバックするマイグレーションの数を指定できます。

cargo loco db down 2

動詞、単数形、複数形

  • **参照**: テーブル名には**単数形**を使用し、`:references`型を使用します。`user:references`（`Users`を参照）、`vote:references`（`Votes`を参照）。`:references:
`も利用可能です。`departing_train:references:trains`（`Trains`を参照）。
  • **カラム名**: 好きなように。`snake_case`を推奨します。
  • **テーブル名**: **複数形、スネークケース**。`users`、`draft_posts`。
  • **マイグレーション名**: ファイル名として使えるものであれば何でも、スネークケースを推奨します。`create_table_users`、`add_vote_id_to_movies`。
  • **モデル名**: 自動的に生成されます。通常、生成される名前はパスカルケースで複数形です。`Users`、`UsersVotes`。

    • 以下に命名規則を示すいくつかの例を示します。

    $ cargo loco generate model movies long_title:string added_by:references:users director:references
    • モデル名は複数形: `movies`
    • 参照の`director`は単数形: `director:references`
    • 参照の`added_by`は単数形、参照されるモデルはモデルであり複数形: `added_by:references:users`
    • カラム名はスネークケース: `long_title:string`

    マイグレーションのオーサリング

    マイグレーションDSLを使用するには、以下の`loco_rs::schema::*`インポートとSeaORM `prelude`があることを確認してください。

    use loco_rs::schema::*;
use sea_orm_migration::prelude::*;

    次に、構造体を作成します。

    #[derive(DeriveMigrationName)]
pub struct Migration;

    そして、マイグレーションを実装します（下記参照）。

    テーブルを作成する

    テーブルを作成し、2つの配列を提供します。(1)カラム (2)参照。

    参照フィールドを作成しない場合は、参照を空にします。

    impl MigrationTrait for Migration {
    async fn up(&self, m: &SchemaManager) -> Result<(), DbErr> {
        create_table(
            m,
            "posts",
            &[
                ("title", ColType::StringNull),
                ("content", ColType::StringNull),
            ],
            &[],
        )
        .await
    }

    async fn down(&self, m: &SchemaManager) -> Result<(), DbErr> {
        drop_table(m, "posts").await
    }
}

    結合テーブルを作成する

    2番目の配列引数に参照を提供します。空文字列 `""` を使用して、参照カラム名を自動生成するように指示します（例：ユーザー参照は、`group_users`テーブルの`user_id`カラムを介して`users`テーブルに接続することを意味します）。

    参照カラム名に特定の名前を示す場合は、空ではない文字列を指定します。

    impl MigrationTrait for Migration {
    async fn up(&self, m: &SchemaManager) -> Result<(), DbErr> {
        create_join_table(m, "group_users", &[], &[("user", ""), ("group", "")]).await
    }

    async fn down(&self, m: &SchemaManager) -> Result<(), DbErr> {
        drop_table(m, "group_users").await
    }
}

    カラムを追加する

    単一のカラムを追加します。1つのマイグレーションで、このようなステートメントをいくつでも使用できます（複数のカラムを追加するため）。

    impl MigrationTrait for Migration {
    async fn up(&self, m: &SchemaManager) -> Result<(), DbErr> {
        add_column(m, "users", "amount", ColType::DecimalLenNull(24,8)).await?;
        Ok(())
    }

    async fn down(&self, m: &SchemaManager) -> Result<(), DbErr> {
        remove_column(m, "users", "amount").await?;
        Ok(())
    }
}

    高度なマイグレーションのオーサリング

    `manager`を直接使用すると、マイグレーションを作成する際により高度な操作にアクセスできます。

    カラムを追加する

      manager
    .alter_table(
        Table::alter()
            .table(Movies::Table)
            .add_column_if_not_exists(integer(Movies::Rating))
            .to_owned(),
    )
    .await

    カラムを削除する

      manager
    .alter_table(
        Table::alter()
            .table(Movies::Table)
            .drop_column(Movies::Rating)
            .to_owned(),
    )
    .await

    インデックスを追加する

    インデックスを追加するためのこのコードの一部をコピーできます

      manager
    .create_index(
        Index::create()
            .name("idx-movies-rating")
            .table(Movies::Table)
            .col(Movies::Rating)
            .to_owned(),
    )
    .await;

    データ修正を作成する

    マイグレーションでデータ修正を作成するのは簡単です。好きなようにSQLステートメントを使用するだけです。

      async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {

    let db = manager.get_connection();
    
    // issue SQL queries with `db`
    // https://www.sea-ql.org/SeaORM/docs/basic-crud/raw-sql/#use-raw-query--execute-interface

    Ok(())
  }

    とはいえ、データ修正をコードで記述するかどうかはあなた次第です。

    • `task` - 高レベルのモデルを使用できる場所
    • `migration` - 構造を変更し、そこから生じるデータを生のSQLで修正できる場所
    • またはアドホックな`playground` - 高レベルのモデルを使用したり、さまざまなことを試したりできる場所

      • バリデーション

      内部では`validator`ライブラリを使用しています。まず、必要な制約でバリデータを作成し、次に`ActiveModel`に`Validatable`を実装します。

      #[derive(Debug, Validate, Deserialize)]
pub struct Validator {
    #[validate(length(min = 2, message = "Name must be at least 2 characters long."))]
    pub name: String,
    #[validate(custom(function = "validation::is_valid_email"))]
    pub email: String,
}

impl Validatable for super::_entities::users::ActiveModel {
    fn validator(&self) -> Box<dyn Validate> {
        Box::new(Validator {
            name: self.name.as_ref().to_owned(),
            email: self.email.as_ref().to_owned(),
        })
    }
}

      `Validatable`は、Locoにどのバリデータを供給し、モデルからそれを構築する方法を指示する方法であることに注意してください。

      これで、コード内で`user.validate()`をシームレスに使用できます。`Ok`の場合はモデルが有効であり、それ以外の場合は`Err(...)`でバリデーションエラーが見つかり、検査できます。

      リレーションシップ

      一対多

      既存の`User`モデルに`Company`を関連付ける方法は次のとおりです。

      $ cargo loco generate model company name:string user:references

      これにより、`Company`に`user_id`フィールドを持つマイグレーションが作成され、それが`User`を参照します。

      多対多

      ここでは、ユーザーと映画を多対多のリンクテーブルで結びつける典型的な「投票」テーブルを作成する方法を示します。モデルジェネレータで特別な`--link`フラグを使用していることに注目してください。

      新しい`Movie`エンティティを作成しましょう。

      $ cargo loco generate model movies title:string

      そして、既存の`User`と、今生成したばかりの`Movie`との間のリンクテーブルを作成し、投票を記録します。

      $ cargo loco generate model --link users_votes user:references movie:references vote:int
    ..
    ..
Writing src/models/_entities/movies.rs
Writing src/models/_entities/users.rs
Writing src/models/_entities/mod.rs
Writing src/models/_entities/prelude.rs
...完了。

      これにより、`user_id`と`movie_id`の両方を含む複合主キーを持つ`UsersVotes`という名前の多対多リンクテーブルが作成されます。正確に2つのIDを持つため、SeaORMはこれを多対多リンクテーブルとして識別し、適切な`via()`リレーションシップを持つエンティティを生成します。

      // User, newly generated entity with a `via` relation at _entities/users.rs

// ..
impl Related<super::movies::Entity> for Entity {
    fn to() -> RelationDef {
        super::users_votes::Relation::Movies.def()
    }
    fn via() -> Option<RelationDef> {
        Some(super::users_votes::Relation::Users.def().rev())
    }
}

      `via()`を使用すると、`find_related`がリンクテーブルの詳細を知ることなく、リンクテーブルをたどることができます。

      設定

      利用可能なモデル設定は、開発、テスト、本番のあらゆる側面を制御し、本番環境での経験から得られた豊富な機能が含まれているため、非常に興味深いものです。

      database:
  # Database connection URI
  uri: 
  # When enabled, the sql query will be logged.
  enable_logging: false
  # Set the timeout duration when acquiring a connection.
  connect_timeout: 500
  # Set the idle duration before closing a connection.
  idle_timeout: 500
  # Minimum number of connections for a pool.
  min_connections: 1
  # Maximum number of connections for a pool.
  max_connections: 1
  # Run migration up when application loaded
  auto_migrate: true
  # Truncate database when application loaded. This is a dangerous operation, make sure that you using this flag only on dev environments or test mode
  dangerously_truncate: false
  # Recreating schema when application loaded.  This is a dangerous operation, make sure that you using this flag only on dev environments or test mode
  dangerously_recreate: false

      これらのフラグを組み合わせることで、より生産性を高めるためのさまざまなエクスペリエンスを作成できます。

      アプリが起動する前にトランケートすることもできます（テストの実行に役立ちます）、またはアプリが起動するときにDB全体を再作成することもできます（統合テストや新しい環境のセットアップに役立ちます）。本番環境では、これらをオフにしたいです（したがって、「dangerously」の部分）。

      シーディング

      Locoには、プロセスを迅速かつ簡単にする組み込みの「シード」機能があります。これは、開発およびテスト環境で頻繁にデータベースをリロードする場合に特に便利です。この機能の使い方は簡単です

      Locoには便利なシード機能が搭載されており、データベースの迅速かつ簡単な再ロードプロセスを効率化します。この機能は、開発およびテスト環境で頻繁にリセットを行う際に特に貴重であることが証明されています。この機能の使用を開始する方法を見ていきましょう。

      新しいシードを作成する

      1. 新しいシードファイルの作成

      `src/fixtures`に移動し、新しいシードファイルを作成します。例：

      src/
  fixtures/
    users.yaml

      このYAMLファイルでは、挿入するデータベースレコードのセットを列挙します。各レコードは、データベースの制約に基づいて必須のデータベースフィールドをすべて含む必要があります。オプションの値は任意です。例えば、以下のようなデータベースDDLがあるとします。

      CREATE TABLE public.users (
	id serial4 NOT NULL,
	email varchar NOT NULL,
	"password" varchar NOT NULL,
	reset_token varchar NULL,
	created_at timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
	CONSTRAINT users_email_key UNIQUE (email),
	CONSTRAINT users_pkey PRIMARY KEY (id)
);

      必須フィールドには、`id`、`password`、`email`、`created_at`が含まれます。リセットトークンは空にできます。マイグレーションコンテンツファイルは次のようになるはずです。

      ---
- id: 1
  email: user1@example.com
  password: "$2b$12$gf4o2FShIahg/GY6YkK2wOcs8w4.lu444wP6BL3FyjX0GsxnEV6ZW"
  created_at: "2023-11-12T12:34:56.789"
- id: 2
  pid: 22222222-2222-2222-2222-222222222222
  email: user2@example.com
  reset_token: "SJndjh2389hNJKnJI90U32NKJ"
  password: "$2b$12$gf4o2FShIahg/GY6YkK2wOcs8w4.lu444wP6BL3FyjX0GsxnEV6ZW"
  created_at: "2023-11-12T12:34:56.789"

      シードを接続する

      次の手順に従って、シードをアプリのフック実装に統合します。

      1. アプリのフック実装に移動します。
      2. シード関数の実装内にシードを追加します。Rustの例を次に示します。
      impl Hooks for App {
    // Other implementations...

    async fn seed(ctx: &AppContext, base: &Path) -> Result<()> {
        db::seed::<users::ActiveModel>(&ctx.db, &base.join("users.yaml").display().to_string()).await?;
        Ok(())
    }
}

      この実装により、シード関数が呼び出されたときにシードが実行されます。アプリケーションの構造と要件に基づいて詳細を調整してください。

      CLIを介したシードの管理

      • **データベースをリセットする**
        シードファイルをインポートする前に、既存のすべてのデータをクリアします。これは、古いデータが残らないように、新しいデータベースの状態から開始したい場合に役立ちます。
      • **データベーステーブルをファイルにダンプする**
        データベーステーブルの内容をファイルにエクスポートします。この機能により、現在のデータベースの状態をバックアップしたり、環境間で再利用するためにデータを準備したりできます。

      シードコマンドにアクセスするには、以下のCLI構造を使用します。

      Seed your database with initial data or dump tables to files

Usage: demo_app-cli db seed [OPTIONS]

      --dump-tables <DUMP_TABLES>  ダンプする特定のテーブルを指定します
