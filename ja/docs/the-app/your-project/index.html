cargo locoによる開発の推進

スターターアプリを作成します:

❯ loco new
✔ ❯ App name? · myapp
✔ ❯ What would you like to build? · Saas App with client side rendering
✔ ❯ Select a DB Provider · Sqlite
✔ ❯ Select your background worker type · Async (in-process tokio async tasks)

🚂 Loco app generated successfully in:
myapp/

- assets: You've selected `clientside` for your asset serving configuration.

Next step, build your frontend:
  $ cd frontend/
  $ npm install && npm run build

次に、アプリにcdして、さまざまなコマンドを試してみてください:

cargo loco --help

The one-person framework for Rust

Usage: demo_app-cli [OPTIONS] <COMMAND>

Commands:
  start       Start an app
  db          Perform DB operations
  routes      Describe all application endpoints
  middleware  Describe all application middlewares
  task        Run a custom task
  jobs        Managing jobs queue
  scheduler   Run the scheduler
  generate    code generation creates a set of files and code templates based on a predefined set of rules
  doctor      Validate and diagnose configurations

Options:
  -e, --environment <ENVIRONMENT>  Specify the environment [default: development]
  -h, --help                       Print help
  -V, --version                    Print version

これでCLIを通じて開発を進めることができます:

$ cargo loco generate model posts
$ cargo loco generate controller posts
$ cargo loco db migrate
$ cargo loco start

テストを実行したりRustを扱ったりするのは、すでにご存知のとおりです:

$ cargo build
$ cargo test

アプリの起動

アプリを実行するには、以下を実行します:

cargo loco start

バックグラウンドワーカー

（config/内の）設定に基づいて、ワーカーはどのように動作するかを認識します:

workers:
  # requires Redis
  mode: BackgroundQueue

  # can also use:
  # ForegroundBlocking - great for testing
  # BackgroundAsync - for same-process jobs, using tokio async

そして今、さまざまな方法で実際のプロセスを実行できます:

  • rr start --worker - ワーカーのみを実行し、バックグラウンドジョブを処理します。これはスケーリングに最適です。1つのサービスアプリをrr startで実行し、その後、好きなマシンに分散された多くのプロセスベースのワーカーをrr start --workerで実行できます。
  • rr start --server-and-worker - サービスとバックグラウンドワーカープロセッサの両方を同じUnixプロセスで実行します。バックグラウンドジョブの実行にはTokioを使用します。これは、あまり費用をかけずに単一サーバーで実行したい場合や、リソースが制約されている場合に最適です。

アプリのバージョンを取得する

アプリはコンパイルされ、その後本番環境にコピーされるため、Locoは2つの重要な操作情報を提供します:

  • このアプリのバージョンは何か、そしてどのGIT SHAからビルドされたのか？ cargo loco version
  • このアプリはどのLocoバージョンに対してコンパイルされたのか？ cargo loco --version

どちらのバージョン文字列も解析可能で安定しているため、統合スクリプトや監視ツールなどで使用できます。

独自のカスタムアプリのバージョン管理スキームは、src/app.rsファイル内のapp_versionフックをオーバーライドすることで形成できます。

スカフォールドジェネレーターを使用する

スカフォールディングは、アプリケーションの主要コンポーネントを生成するための効率的で迅速な方法です。スカフォールディングを利用することで、新しいリソースのモデル、ビュー、コントローラーを一度にすべて作成できます。

スカフォールドコマンドを参照:

Generates a CRUD scaffold, model and controller

Usage: demo_app-cli generate scaffold [OPTIONS] <NAME> [FIELDS]...

Arguments:
  <NAME>       Name of the thing to generate
  [FIELDS]...  Model fields, eg. title:string hits:int

Options:
  -k, --kind <KIND>                The kind of scaffold to generate [possible values: api, html, htmx]
      --htmx                       Use HTMX scaffold
      --html                       Use HTML scaffold
      --api                        Use API scaffold
  -e, --environment <ENVIRONMENT>  Specify the environment [default: development]
  -h, --help                       Print help
  -V, --version                    Print version

まず、Postリソースのスカフォールドを生成することから始めます。これは単一のブログ投稿を表します。これを行うには、ターミナルを開き、次のコマンドを入力します。

cargo loco generate scaffold posts name:string title:string content:text --api

スカフォールド生成コマンドは、--templateフラグをスカフォールドコマンドに追加することで、API、HTML、またはHTMXをサポートします。

アプリの設定

デフォルトでは、Locoは設定ファイルをconfig/ディレクトリに保存します。3つの環境に対して事前定義された設定を提供します:

config/
  development.yaml
  production.yaml
  test.yaml

環境は、以下に基づいて自動的に選択されます:

  • コマンドラインフラグ: cargo loco start --environment production（指定しない場合、以下にフォールバック）
  • LOCO_ENV または RAILS_ENV または NODE_ENV

何も指定されない場合、デフォルト値はdevelopmentです。

Locoフレームワークは、デフォルト環境に加えてカスタム環境もサポートしています。カスタム環境を追加するには、前述の例で使用されている環境識別子に一致する名前で設定ファイルを作成します。

デフォルトの設定パスのオーバーライド

カスタム設定ディレクトリを使用するには、LOCO_CONFIG_FOLDER環境変数に目的のフォルダパスを設定します。これにより、locoはデフォルトのconfig/フォルダではなく、指定されたディレクトリから設定ファイルをロードするよう指示されます。

設定におけるプレースホルダー / 変数

設定ファイルに値を注入することが可能です。この例では、NODE_PORT環境変数からポート値を取得しています:

# config/development.yaml
# every configuration file is a valid Tera template
server:
  # Port on which the server will listen. the server binding is 0.0.0.0:{PORT}
  port:  {{ get_env(name="NODE_PORT", default=5150) }}
  # The UI hostname or IP address that mailers will point to.
  host: http://localhost
  # Out of the box middleware configuration. to disable middleware you can changed the `enable` field to `false` of comment the middleware block

get_env関数はTeraテンプレートエンジンの一部です。Teraのドキュメントを参照して、さらに使用できるものを確認してください。

例

例えば、「qa」環境を追加したいとします。configフォルダにqa.yamlファイルを作成します:

config/
  development.yaml
  production.yaml
  test.yaml
  qa.yaml

「qa」環境を使用してアプリケーションを実行するには、次のコマンドを実行します:

LOCO_ENV=qa cargo loco start

設定

設定ファイルにはLocoアプリを設定するためのノブが含まれています。settings:セクションを使用してカスタム設定を持つこともできます。config/development.yamlにsettings:セクションを追加します

settings:
  allow_list:
    - google.com
    - apple.com

これらの設定はctx.config.settingsにserde_json::Valueとして現れます。structを追加することで、厳密に型付けされた設定を作成できます:

// put this in src/common/settings.rs
#[derive(Serialize, Deserialize, Default, Debug)]
pub struct Settings {
    pub allow_list: Option<Vec<String>>,
}

impl Settings {
    pub fn from_json(value: &serde_json::Value) -> Result<Self> {
        Ok(serde_json::from_value(value.clone())?)
    }
}

次に、このようにどこからでも設定にアクセスできます:

// in controllers, workers, tasks, or elsewhere,
// as long as you have access to AppContext (here: `ctx`)

if let Some(settings) = &ctx.config.settings {
    let settings = common::settings::Settings::from_json(settings)?;
    println!("allow list: {:?}", settings.allow_list);
}

サーバー

以下に、server:以下のインターフェース（リスニングなど）パラメータの詳細な説明を示します:

  • port: 名前が示すように、主にロードバランサーの背後などでポートを変更するために使用されます。

  • binding: IPインターフェースが「バインド」するものを変更するために使用されます。主に、nginxのようなロードバランサーの背後にいる場合、ローカルアドレスにバインドします（LBもそこにある場合）。ただし、「ワールド」（0.0.0.0）にバインドすることもできます。設定ファイルまたはCLI（-bフラグを使用）を介してbinding:フィールドを設定できます。これはRailsが行っていることと同じです。

  • host: - 「可視性」のユースケースや帯域外のユースケースに使用されます。たとえば、現在のサーバーホスト（ドメイン名など）を表示したい場合があります。これは可視性のために役立ちます。また、Eメールの場合のように、サーバーアドレスが「帯域外」である場合があります。これは、Gmailアカウントを開いてあなたのEメールを受信したときに、内部の「ホスト」アドレスではなく、外部のアドレスまたは可視のアドレス（公式ドメイン名など）のように見えるものをクリックする必要があることを意味します（「http://127.0.0.1/account/verify」を指すEメールリンクを想像してみてください）。

ロガー

YAMLファイルのlogger:セクションにあるコメントアウトされたフィールド以外に、さらにいくつかのコンテキストを以下に示します:

  • logger.pretty_backtrace - 素晴らしい開発体験のために、ノイズのないカラフルなバックトレースを表示します。これは、プロセス環境に強制的にRUST_BACKTRACE=1を設定し、特定のエラーで（コストのかかる）バックトレースキャプチャを有効にします。開発環境では有効にし、本番環境では無効にしてください。本番環境で必要な場合は、コマンドラインで一時的にRUST_BACKTRACE=1を使用して表示します。
