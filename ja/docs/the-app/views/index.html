ビュー

Loco では、Web リクエストの処理はコントローラ、モデル、ビューに分かれています。

  • コントローラ はリクエストのペイロード解析を処理し、その後、モデルに制御を渡します
  • モデル は、主にデータベースとの通信と、必要に応じた CRUD 操作の実行を扱います。また、すべてのビジネスロジックとドメインロジック、および操作をモデル化します。
  • ビュー は、クライアントに送り返される最終的なレスポンスの組み立てとレンダリングの責任を負います。

JSON レスポンスであるJSON ビュー、またはテンプレートビューエンジンによって動作し最終的に HTML レスポンスとなるテンプレートビューを選択できます。両方を組み合わせることも可能です。

これは、Rails の JSON である `jbuilder` ビューや、HTML である通常のビューと似ていますが、LOCO では JSON ファーストであることに焦点を当てています。

JSON ビュー

例として、ユーザーログインを処理するエンドポイントがあります。ユーザーが有効な場合、user モデルを LoginResponse ビュー（JSON ビュー）に渡し、レスポンスを返すことができます。

3つのステップがあります。

  1. リクエストを解析し、受け入れる
  2. ドメインオブジェクトの作成: モデル
  3. ドメインモデルを、最終的なレスポンスを整形するビューオブジェクトに渡す

以下の Rust コードは、ユーザーログインリクエストを処理するコントローラを表しており、レスポンスの整形を LoginResponse に委ねています。

use crate::{views::auth::LoginResponse};
async fn login(
    State(ctx): State<AppContext>,
    Json(params): Json<LoginParams>,
) -> Result<Response> {
    // Fetching the user model with the requested parameters
    // let user = users::Model::find_by_email(&ctx.db, &params.email).await?;

    // Formatting the JSON response using LoginResponse view
    format::json(LoginResponse::new(&user, &token))
}

一方、LoginResponse は serde によって動作するレスポンス整形ビューです。

use serde::{Deserialize, Serialize};

use crate::models::_entities::users;

#[derive(Debug, Deserialize, Serialize)]
pub struct LoginResponse {
    pub token: String,
    pub pid: String,
    pub name: String,
}

impl LoginResponse for LoginResponse {
    #[must_use]
    pub fn new(user: &users::Model, token: &String) -> Self {
        Self {
            token: token.to_string(),
            pid: user.pid.to_string(),
            name: user.name.clone(),
        }
    }
}

テンプレートビュー

ユーザーに HTML を返したい場合、サーバーサイドテンプレートを使用します。これは、Ruby の erb や Node の ejs、あるいは PHP の動作と似ています。

サーバーサイドテンプレートのレンダリングには、人気のある Tera テンプレートエンジンに基づいた組み込みの TeraView エンジンを提供しています。

このエンジンを使用するには、`initializers/view_engine.rs` に `ViewEngineInitializer` があり、それが `app.rs` でも指定されていることを確認する必要があります。SaaS スターターを使用した場合、これはすでに構成されています。

Tera ビューエンジンは、新しい assets/ フォルダからリソースを取得します。以下に構造の例を示します。

新しいビューの作成

まず、テンプレートを作成します。この例では、assets/views/home/hello.html に Tera テンプレートを追加します。assets/ はプロジェクトのルート（src/ と config/ の隣）に配置されることに注意してください。

<html><body>
find this tera template at <code>assets/views/home/hello.html</code>: 
<br/>
<br/>
{{ /* t(key="hello-world", lang="en-US") */ }}, 
<br/>
{{ /* t(key="hello-world", lang="de-DE") */ }}

</body></html>

次に、このテンプレートをカプセル化するために、src/views/dashboard.rs に厳密に型付けされた view を作成します。

// src/views/dashboard.rs
use loco_rs::prelude::*;

pub fn home(v: impl ViewRenderer) -> Result<impl IntoResponse> {
    format::render().view(&v, "home/hello.html", data!({}))
}

そして、src/views/mod.rs に追加します。

pub mod dashboard;

次に、コントローラに移動してビューを使用します。

// src/controllers/dashboard.rs
use loco_rs::prelude::*;

use crate::views;

pub async fn render_home(ViewEngine(v): ViewEngine<TeraView>) -> Result<impl IntoResponse> {
    views::dashboard::home(v)
}

pub fn routes() -> Routes {
    Routes::new().prefix("home").add("/", get(render_home))
}

最後に、新しいコントローラのルートを src/app.rs に登録します。

pub struct App;
#[async_trait]
impl Hooks for App {
    // omitted for brevity

    fn routes(_ctx: &AppContext) -> AppRoutes {
        AppRoutes::with_default_routes()
            .add_route(controllers::auth::routes())
            // include your controller's routes here
            .add_route(controllers::dashboard::routes())
    }

上記すべてを完了すると、cargo loco routes を実行したときに新しいルートが表示されるはずです。

$ cargo loco routes

[GET] /_ping
[POST] /api/auth/forgot
[POST] /api/auth/login
[POST] /api/auth/register
[POST] /api/auth/reset
[POST] /api/auth/verify
[GET] /api/auth/current
[GET] /home              <-- the corresponding URL for our new view

仕組み

  • ViewEngine は、loco_rs::prelude::* を介して利用できるエクストラクタです。
  • TeraView は Loco に付属する Tera ビューエンジンで、これも loco_rs::prelude::* を介して利用できます。
  • コントローラは、リクエストの取得、モデルロジックの呼び出し、そしてモデルとその他のデータをビューに提供することに専念し、ビューがどのように動作するかは気にしません。
  • views::dashboard::home は不透明な呼び出しであり、ビューがどのように動作するか、またはバイトがどのようにブラウザに到達するかという詳細を隠蔽します。これは良いことです。
  • ビューエンジンを交換したい場合でも、ここでのカプセル化は魔法のように機能します。エクストラクタの型を ViewEngine<Foobar> に変更するだけで、v が最終的には単なる ViewRenderer トレイトであるため、すべてが動作します。

静的アセット

静的アセットを提供し、それをビューテンプレート内で参照したい場合は、静的ミドルウェアをこのように設定できます。

static:
  enable: true
  must_exist: true
  precompressed: false
  folder:
    uri: "/static"
    path: "assets/static"
  fallback: "assets/static/404.html"

テンプレート内では、静的リソースをこのように参照できます。

<img src="/static/image.png"/>

ただし、静的ミドルウェアが機能するには、デフォルトのフォールバックが無効になっていることを確認してください。

fallback:
  enable: false

Tera ビューエンジンのカスタマイズ

Tera ビューエンジンは以下の設定で提供されます。

  • テンプレートの読み込みと場所: assets/**/*.html
  • Tera ビューエンジンに国際化 (i18n) が設定されているため、テンプレートで翻訳関数: t(..) を使用できます。

i18n ライブラリの設定の詳細を変更したい場合は、src/initializers/view_engine.rs を編集できます。

イニシャライザを編集することで、以下を行うことができます。

  • カスタム Tera 関数の追加
  • i18n ライブラリの削除
  • Tera または i18n ライブラリの設定変更
  • 新しい、またはカスタムの Tera インスタンス（異なるバージョンでも可）の提供

独自のビューエンジンの使用

Tera をビューエンジンとして気に入らない場合や、Handlebars などを使用したい場合は、独自のカスタムビューエンジンを非常に簡単に作成できます。

ダミーの「Hello」ビューエンジンの例を挙げます。これは常にhelloという単語を返すビューエンジンです。

// src/initializers/hello_view_engine.rs
use axum::{Extension, Router as AxumRouter};
use async_trait::async_trait;
use loco_rs::{
    app::{AppContext, Initializer},
    controller::views::{ViewEngine, ViewRenderer},
    Result,
};
use serde::Serialize;

#[derive(Clone)]
pub struct HelloView;
impl ViewRenderer for HelloView {
    fn render<S: Serialize>(&self, _key: &str, _data: S) -> Result<String> {
        Ok("hello".to_string())
    }
}

pub struct HelloViewEngineInitializer;
#[async_trait]
impl Initializer for HelloViewEngineInitializer {
    fn name(&self) -> String {
        "custom-view-engine".to_string()
    }

    async fn after_routes(&self, router: AxumRouter, _ctx: &AppContext) -> Result<AxumRouter> {
        Ok(router.layer(Extension(ViewEngine::from(HelloView)))
    }
}

これを使用するには、src/app.rs のフックに追加する必要があります。

// src/app.rs
// add your custom "hello" view engine in the `initializers(..)` hook
impl Hooks for App {
    // ...
    async fn initializers(_ctx: &AppContext) -> Result<Vec<Box<dyn Initializer>>> {
        Ok(vec![
            // ,.----- add it here
            Box::new(initializers::hello_view_engine::HelloViewEngineInitializer),
        ])
    }
    // ...
