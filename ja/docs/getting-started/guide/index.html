Locoガイド

ガイドの前提条件

これは「遠回り」のチュートリアルです。意図的に長く掘り下げており、手動で構築する方法と、ジェネレータを使用して自動的に構築する方法の両方を示しているので、構築スキルと仕組みの両方を学ぶことができます。

名前の由来は？

「Loco」という名前は、Railsへの敬意を表してlocomotive（機関車）から来ており、locoはlocomotiveよりも入力しやすいためです:-)。また、一部の言語では「狂った」という意味もありますが、それは元の意図ではありませんでした（あるいは、RustでRailsを構築するのは狂っているのでしょうか？それは時が経てばわかることです！）。

どのくらいのRustの知識が必要ですか？

Rustについては、初心者レベルから中級初心者のレベルまで慣れ親しんでいる必要があります。Rustプロジェクトのビルド、テスト、実行方法を知っており、clap、regex、tokio、axumなどの人気ライブラリやその他のウェブフレームワークを使用した経験があれば十分です。Locoには、その仕組みを知る必要のある複雑すぎるライフタイムのねじれや魔法のようなマクロはありません。

Locoとは？

LocoはRailsに強く影響を受けています。RailsとRustの両方を知っていれば、すぐに馴染めるでしょう。Railsしか知らず、Rustが初めてでも、Locoは新鮮に感じられるはずです。Railsの知識があることは前提としていません。

私たちはRailsが非常に優れていると考えているため、このガイドもRailsのガイドから強く影響を受けています。

LocoはRust用のWebまたはAPIフレームワークです。また、開発者向けの生産性スイートでもあり、趣味のプロジェクトや次のスタートアップを構築する際に必要なすべてが含まれています。Railsから強く影響を受けています。

  • MVCモデルのバリアントがあり、オプションのパラドックスを取り除きます。使用する抽象化について学術的な決定を下すのではなく、アプリの構築に集中できます。
  • ファットモデル、スリムコントローラ。モデルはほとんどのロジックとビジネス実装を含むべきであり、コントローラはHTTPを理解し、パラメータをやり取りする軽量なルータであるべきです。
  • コマンドライン駆動で勢いとフローを維持します。コピー＆ペーストやゼロからのコーディングよりも生成を重視します。
  • すべてのタスクは「インフラストラクチャ対応」であり、コードをプラグインして接続するだけです。コントローラ、モデル、ビュー、タスク、バックグラウンドジョブ、メーラーなど。
  • 設定より規約：決定はすでにあなたのために行われています。フォルダ構造、設定の形式と値、アプリの接続方法が、アプリの動作とあなたの最も効果的な作業に影響します。

新しいLocoアプリの作成

このガイドに従って段階的に「ボトムアップ」で学習することも、代わりに「トップダウン」で素早く概要を把握するためにツアーにジャンプすることもできます。

「こんにちは」、Loco

サービスに簡単な「hello」応答を追加しましょう。

$ cargo loco generate controller guide --api

injected: "src/controllers/mod.rs"
injected: "src/app.rs"
added: "tests/requests/guide.rs"
injected: "tests/requests/mod.rs"

生成されたコントローラの本体です:

#![allow(clippy::missing_errors_doc)]
#![allow(clippy::unnecessary_struct_initialization)]
#![allow(clippy::unused_async)]
use loco_rs::prelude::*;
use axum::debug_handler;

#[debug_handler]
pub async fn index(State(_ctx): State<AppContext>) -> Result<Response> {
    format::empty()
}

pub fn routes() -> Routes {
    Routes::new()
        .prefix("api/guides/")
        .add("/", get(index))
}

インデックスハンドラの本体を変更します:

// replace
    format::empty()
// with this
    format::text("hello")

サーバーを起動:

cargo loco start

さあ、テストしてみましょう:

$ curl localhost:5150/api/guides
hello

Locoには強力なジェネレータがあり、アプリ構築時の生産性を10倍向上させ、勢いを加速させます。

もし少しの間、お楽しみになりたいなら、「苦労して学ぶ」方法で新しいコントローラを手動で追加してみましょう。

home.rsというファイルと、mod.rsにpub mod home;という行を追加します:

src/
  controllers/
    auth.rs
    home.rs      <--- このファイルを追加
    users.rs
    mod.rs       <--- ここでモジュールを 'pub mod home;' します

次に、helloルートを設定します。これがhome.rsの内容です:

// src/controllers/home.rs
use loco_rs::prelude::*;

// _ctx contains your database connection, as well as other app resource that you'll need
async fn hello(State(_ctx): State<AppContext>) -> Result<Response> {
    format::text("ola, mundo")
}

pub fn routes() -> Routes {
    Routes::new().prefix("home").add("/hello", get(hello))
}

最後に、この新しいコントローラのルートをapp.rsに登録します:

src/
  controllers/
  models/
  ..
  app.rs   <---- ここを見てください

routes()に以下を追加します:

// in src/app.rs
#[async_trait]
impl Hooks for App {
    ..
    fn routes() -> AppRoutes {
        AppRoutes::with_default_routes()
            .add_route(controllers::guide::routes())
            .add_route(controllers::auth::routes())
            .add_route(controllers::home::routes()) // <--- これを追加
    }

これで完了です。サーバーを終了し、再度起動します:

cargo loco start

そして/home/helloにアクセスします:

$ curl localhost:5150/home/hello
ola, mundo

すべてのルートは次で確認できます:

$ cargo loco routes

  ..

[POST] /api/auth/register
[POST] /api/auth/reset
[POST] /api/auth/verify
[GET] /home/hello      <---- これが新しいルートです！

SaaSスターターは、クライアントサイド対応であり、スキャフォールディングで--apiオプションを使用しているため、ルートを/apiの下に保持します。
React Routerのようなクライアントサイドルーティングを使用する場合、バックエンドルートとクライアントルートを分離したいと考えます。ブラウザは/homeを使用しますが、バックエンドルートである/api/homeは使用せず、クライアントから/api/homeを問題なく呼び出すことができます。ただし、/_healthと/_pingのルートは例外で、ルートのままです。

MVCとあなた

従来のMVC（Model-View-Controller）はデスクトップUIプログラミングのパラダイムに由来しています。しかし、そのウェブサービスへの適用可能性から急速に採用されました。MVCの黄金時代は2010年代初頭頃で、それ以来、他の多くのパラダイムやアーキテクチャが登場しました。

MVCはプロジェクトを簡素化するための非常に強力な原則でありアーキテクチャであり、Locoもそれに従っています。

ウェブサービスやAPIはHTMLやUI応答を生成しないため、ビューの概念を持ちませんが、安定した安全なサービスとAPIは実際にビューの概念を持っていると主張します。それは、シリアライズされたデータ、その形式、互換性、およびバージョンです。

// a typical loco app contains all parts of MVC

src/
  controllers/
    users.rs
    mod.rs
  models/
    _entities/
      articles.rs  <-- DBスキーマから同期されています。編集しないでください
      mod.rs
      prelude.rs
      users.rs
    articles.rs   <-- あなたのために生成されました。ロジックはここに入力します。
    mod.rs
    users.rs
  views/
    users.rs
    mod.rs

これは重要な認知原則です。そして、この原則は、安全で互換性のあるAPIレスポンスを作成するには、それらを独立して管理される別の「もの」として扱う必要があると主張しています。これがLocoにおけるMVCの「V」の理由です。

Locoのモデルは、Railsと同じセマンティクス（意味）を持っています。ファットモデル、スリムコントローラです。これは、何かを構築したいときはいつでもモデルにアクセスすることを意味します。

モデルの生成

Locoのモデルはデータと機能を表現します。通常、データはデータベースに保存されます。アプリケーションのほとんどの、あるいはすべてのビジネスプロセスは、モデル（Active Recordとして）またはいくつかのモデルのオーケストレーションとしてコーディングされます。

Articleという新しいモデルを作成しましょう:

$ cargo loco generate model article title:string content:text

added: "migration/src/m20231202_173012_articles.rs"
injected: "migration/src/lib.rs"
injected: "migration/src/lib.rs"
added: "tests/models/articles.rs"
injected: "tests/models/mod.rs"

データベースマイグレーション

スキーマの一貫性を保つには、マイグレーションを使用します。マイグレーションはデータベース構造に対する単一の変更であり、テーブルの追加、変更、インデックスの作成などを含むことができます。

// this was generated into `migrations/` from the command:
//
// $ cargo loco generate model article title:string content:text
//
// it is automatically applied by Loco's migrator framework.
// you can also apply it manually using the command:
//
// $ cargo loco db migrate
//
#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                table_auto_tz(Articles::Table)
                    .col(pk_auto(Articles::Id))
                    .col(string_null(Articles::Title))
                    .col(text(Articles::Content))
                    .to_owned(),
            )
            .await
    }

    async fn down(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .drop_table(Table::drop().table(Articles::Table).to_owned())
            .await
    }
}

新しいデータベースにマイグレーションを連続して適用することで、完全なデータベースを再構築できます。これはLocoのマイグレータ（SeaORMから派生）によって自動的に行われます。

新しいモデルを生成する際、Locoは以下を行います:

  • 新しい「アップ」データベースマイグレーションを生成
  • マイグレーションを適用
  • データベース構造からエンティティを反映し、_entitiesコードを再生成

新しいモデルは、models/_entities/にデータベース構造から同期されたエンティティとして見つかります:

src/models/
├── _entities
│   ├── articles.rs  <-- DBスキーマから同期されています。編集しないでください
│   ├── mod.rs
│   ├── prelude.rs
│   └── users.rs
├── articles.rs   <-- あなたのために生成されました。ロジックはここに入力します。
├── mod.rs
└── users.rs

$ cargo loco generate controller articles --api
added: "src/controllers/articles.rs"
injected: "src/controllers/mod.rs"
injected: "src/app.rs"
added: "tests/requests/articles.rs"
injected: "tests/requests/mod.rs"

src/controllers/articles.rsを編集:

// this is src/controllers/articles.rs

#![allow(clippy::unused_async)]
use loco_rs::prelude::*;
use serde::{Deserialize, Serialize};

use crate::models::_entities::articles::{ActiveModel, Entity, Model};

#[derive(Clone, Debug, Serialize, Deserialize)]
pub struct Params {
    pub title: Option<String>,
    pub content: Option<String>,
}

impl Params {
    fn update(&self, item: &mut ActiveModel) {
        item.title = Set(self.title.clone());
        item.content = Set(self.content.clone());
    }
}

async fn load_item(ctx: &AppContext, id: i32) -> Result<Model> {
    let item = Entity::find_by_id(id).one(&ctx.db).await?;
    item.ok_or_else(|| Error::NotFound)
}

pub async fn list(State(ctx): State<AppContext>) -> Result<Response> {
    format::json(Entity::find().all(&ctx.db).await?)
}

pub async fn add(State(ctx): State<AppContext>, Json(params): Json<Params>) -> Result<Response> {
    let mut item: ActiveModel = Default::default();
    params.update(&mut item);
    let item = item.insert(&ctx.db).await?;
    format::json(item)
}

pub async fn update(
    Path(id): Path<i32>,
    State(ctx): State<AppContext>,
    Json(params): Json<Params>,
) -> Result<Response> {
    let item = load_item(&ctx, id).await?;
    let mut item = item.into_active_model();
    params.update(&mut item);
    let item = item.update(&ctx.db).await?;
    format::json(item)
}

pub async fn remove(Path(id): Path<i32>, State(ctx): State<AppContext>) -> Result<Response> {
    load_item(&ctx, id).await?.delete(&ctx.db).await?;
    format::empty()
}

pub async fn get_one(Path(id): Path<i32>, State(ctx): State<AppContext>) -> Result<Response> {
    format::json(load_item(&ctx, id).await?)
}

pub fn routes() -> Routes {
    Routes::new()
        .prefix("api/articles")
        .add("/", get(list))
        .add("/", post(add))
        .add("/{id}", get(get_one))
        .add("/{id}", delete(remove))
        .add("/{id}", patch(update))
}

いくつか注意点があります:

  • Paramsは強く型付けされた必須パラメータのデータホルダーであり、Railsのストロングパラメータに似ていますが、より安全です。
  • Path(id): PathはURLから:idコンポーネントを抽出します。
  • エクストラクタの順序は重要であり、axumのドキュメント（パラメータ、状態、ボディ）に従います。
  • 常にload_itemヘルパー関数を作成し、すべての単一アイテムルートで使用する方が良いです。
  • use loco_rs::prelude::*はコントローラを構築するために必要なすべてを取り込みますが、パラメータのためにcrate::models::_entities::articles::{ActiveModel, Entity, Model}とSerialize, Deserializeもインポートすることに注意してください。
エクストラクタの順序は重要です。順序を変更するとコンパイルエラーにつながる可能性があります。ハンドラに#[debug_handler]マクロを追加すると、より良いエラーメッセージを出力することで役立ちます。エクストラクタに関する詳細情報は、axumのドキュメントで確認できます。
