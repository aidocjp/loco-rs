Axum vs Loco

注：LocoはAxumをベースにしており、「バッテリー付属のAxum」であるため、AxumのコードをLocoに移行するのは非常に簡単です。

私たちはrealworld-axum-sqlxというAxumベースのアプリを調査します。このアプリは、API、実際のデータベース、実世界のシナリオ、および構成やロギングなどの実世界の運用要件を使用して、現実のプロジェクトを説明しようとするものです。

realworld-axum-sqlxを細かく分解することで、AxumからLocoに移行することで、ほとんどのコードがすでに記述されており、より良いベストプラクティス、より良い開発体験、統合されたテスト、コード生成、そしてより迅速なアプリ開発が可能になることを示します。

この分解を理解して、ご自身のAxumベースのアプリをLocoに移行する方法を理解することもできます。ご質問がある場合は、議論までお問い合わせいただくか、緑色の招待ボタンをクリックしてDiscordに参加してください。

main

Axumを使用する場合、独自のmain関数を作成して、アプリのすべてのコンポーネントをセットアップし、ルーターを取得し、ミドルウェアを追加し、コンテキストを設定し、最終的にソケットでlistenを設定する必要があります。

これは多くの手作業であり、エラーが発生しやすい作業です。

Locoでは：

  • 構成で必要なミドルウェアをオン/オフします
  • cargo loco startを使用します。mainファイルは一切必要ありません
  • 本番環境では、your_appという名前のコンパイル済みバイナリを実行します

Locoへの移行

  • Locoのconfig/で必要なミドルウェアを設定します
server:
  middlewares:
    limit_payload:
      body_limit: 5mb
  # .. more middleware below ..
  • Locoのconfig/でサービス提供ポートを設定します
server:
  port: 5150

評価

  • 記述するコードなし、必要な場合を除き、手動でmain関数を記述する必要はありません
  • すぐに使えるベストプラクティス、すべてのLocoアプリで共通のメインファイルのベストプラクティスが得られます
  • 変更が簡単、ミドルウェアを削除/追加してテストしたい場合、構成でスイッチを切り替えるだけで、再ビルドは不要です

環境変数

realworld axumのコードベースはdotenvを使用しており、mainで明示的にロードする必要があります：

 dotenv::dotenv().ok();

そして、.envファイルが利用可能で、維持され、ロードされる必要があります：

DATABASE_URL=postgresql://postgres:{password}@localhost/realworld_axum_sqlx
HMAC_KEY={random-string}
RUST_LOG=realworld_axum_sqlx=debug,tower_http=debug

これはプロジェクトに付属しているサンプルファイルであり、手動でコピーして編集する必要があり、多くの場合非常にエラーが発生しやすいです。

Locoへの移行

Loco: 標準のconfig/[stage].yaml構成を使用し、get_envを使って環境から特定の値をロードします：

# config/development.yaml

# Web server configuration
server:
  # Port on which the server will listen. the server binding is 0.0.0.0:{PORT}
  port:  {{ get_env(name="NODE_PORT", default=5150) }}

この構成は厳密に型付けされており、データベースURL、ロガーレベル、フィルタリングなど、最もよく使用される値を含んでいます。推測したり、車輪の再発明をする必要はありません。

評価

  • コーディング不要、Locoに移行すると、記述するコードが少なくなります
  • 可動部分が少ない、Axumのみを使用する場合、環境変数に加えて構成が必要ですが、これはLocoで無料で得られます

データベース

Axumのみを使用する場合、通常、接続、プールを設定し、それをルートで利用できるようにする必要があります。以下は、通常main.rsに記述するコードです：

    let db = PgPoolOptions::new()
        .max_connections(50)
        .connect(&config.database_url)
        .await
        .context("could not connect to database_url")?;

その後、この接続を手動で配線する必要があります

 .layer(AddExtensionLayer::new(ApiContext {
                config: Arc::new(config),
                db,
            }))

Locoへの移行

Locoでは、config/フォルダーでプール設定値を設定するだけです。デフォルト値はすでに自動で設定されるため、手動で行う必要はありませんが、もし設定したい場合は、以下のようになります：

database:
  enable_logging: false
  connect_timeout: 500
  idle_timeout: 500
  min_connections: 1
  max_connections: 1

評価

  • 記述するコードなし - DBプールに適切な値を選択したり、誤って設定したりする危険を避けることができます
  • 変更が簡単 - 本番環境でさまざまな負荷の下で異なる値を試したい場合、Axumのみでは再コンパイル、再デプロイが必要です。Locoでは、構成を設定してプロセスを再起動できます。

ロギング

アプリ全体で、ロギングに関するコードを手動で記述する必要があります。どれを選びますか？tracingそれともslog？ロギングですか、それともトレースですか？どちらが良いですか？

リアルワールドAxumプロジェクトには、以下のようなものがあります。サービス提供時：

  // Enables logging. Use `RUST_LOG=tower_http=debug`
  .layer(TraceLayer::new_for_http()),

そしてmain内：

    // Initialize the logger.
    env_logger::init();

そして、さまざまな場所でのアドホックロギング：

  log::error!("SQLx error: {:?}", e);

Locoへの移行

Locoでは、これらの難しい問題にすでに対処し、多層的なロギングとトレースを提供しています：

  • フレームワーク内部で
  • ルーターで設定
  • 低レベルDBロギングとトレース
  • タスク、バックグラウンドジョブなど、Locoのすべてのコンポーネントは同じ機能を使用します

そして私たちはtracingを選択しました。これにより、あらゆるRustライブラリがログに均一に「ストリーム」できます。

しかし、デフォルトでは、知らないライブラリでログがあふれないように、スマートフィルターを作成することも確認しました。

ロガーはconfig/で設定できます：

logger:
  enable: true
  pretty_backtrace: true
  level: debug
  format: compact

評価

  • 記述するコードなし - セットアップコードも、決定も不要です。私たちはあなたのアプリのために最善の決定を下したので、あなたはアプリのコードをより多く書くことができます。
  • より速く構築 - 必要なものだけのトレースが得られます。カラフルで文脈に沿った、ノイズのないエラーバックトレースが得られ、デバッグが容易になります。本番環境に合わせてフォーマットやレベルを変更できます。
  • 変更が簡単 - 本番環境でさまざまな負荷の下で異なる値を試したい場合、Axumのみでは再コンパイル、再デプロイが必要です。Locoでは、構成を設定してプロセスを再起動できます。

ルーティング

AxumからLocoへのルートの移行は、実際にはドロップインで可能です。LocoはネイティブのAxumルーターを使用しています。

ルートの一覧表示や情報のような機能が必要な場合、Axumルーターに変換されるネイティブのLocoルーターを使用することも、独自のAxumルーターを使用することもできます。

Locoへの移行

1対1の完全なコピー＆ペースト体験をしたい場合は、Axumのルートをコピーして、Locoのafter_routes()フックにルーターをプラグインするだけです：

  async fn after_routes(router: AxumRouter, _ctx: &AppContext) -> Result<AxumRouter> {
      // use AxumRouter to mount your routes and return an AxumRouter
  }

Locoにルートのメタデータ情報（後で役立つことがあります）を理解させたい場合は、各コントローラーのroutes()関数を次のように記述してください：

// this is what people usually do using Axum only
pub fn router() -> Router {
  Router::new()
        .route("/auth/register", post(create_user))
        .route("/auth/login", post(login_user))
}

// this is how it looks like using Loco (notice we use `Routes` and `add`)
pub fn routes() -> Routes {
  Routes::new()
      .add("/auth/register", post(create_user))
      .add("/auth/login", post(login_user))
}

評価

  • ドロップイン互換性 - LocoはAxumを使用し、その構成要素をすべてそのまま保持しているため、既存のAxumコードを労力なしでそのまま使用できます。
  • ルートメタデータを無料で - Axumルーターの1つの欠点は、現在構成されているルートを記述する機能であり、これはリスト表示や自動OpenAPIスキーマ生成に使用できます。Locoには、これをサポートするための小さなメタデータ層があります。Routesを使用すると、異なるシグネチャがすべてAxumルーターと互換性を保ちながら、この機能を無料で利用できます。
