ワーカー

Locoは、バックグラウンドジョブに対して以下のオプションを提供します。

  • Redisバックエンド（sidekiq-rsによる）
  • Postgresバックエンド（独自の実装）
  • SQLiteバックエンド（独自の実装）
  • Tokio-asyncベース（同一プロセス、イベント駆動スレッドベースのバックグラウンドジョブ）

RailsのActiveJobと同様に、実際のバックグラウンドキューの実装を知らなくてもジョブをキューに入れて実行できるため、設定を簡単に変更するだけでコードを変更せずに切り替えが可能です。

Async vs キュー

新しいアプリを生成した際、ワーカーのデフォルトasync設定を選択したかもしれません。これは、ワーカーがTokioの非同期プールでジョブを起動し、同じ実行中のサーバー内で適切なバックグラウンドプロセスを提供することを意味します。

サーバー間で負荷を分散させるため、キューに支えられた別のプロセスでジョブを実行するように設定したい場合があります。

まず、BackgroundQueueに切り替えます。

# Worker Configuration
workers:
  # specifies the worker mode. Options:
  #   - BackgroundQueue - Workers operate asynchronously in the background, processing queued.
  #   - ForegroundBlocking - Workers operate in the foreground and block until tasks are completed.
  #   - BackgroundAsync - Workers operate asynchronously in the background, processing tasks with async capabilities.
  mode: BackgroundQueue

次に、Redisベースのキューバックエンドを設定します。

queue:
  kind: Redis
  # Redis connection URI.
  uri: ""
  # Dangerously flush all data.
  dangerously_flush: false
  # represents the number of tasks a worker can handle simultaneously.
  num_workers: 2

またはPostgresベースのキューバックエンドです。

queue:
  kind: Postgres
  # Postgres Queue connection URI.
  uri: ""
  # Dangerously flush all data.
  dangerously_flush: false
  # represents the number of tasks a worker can handle simultaneously.
  num_workers: 2

またはSQLiteベースのキューバックエンドです。

queue:
  kind: Sqlite
  # SQLite Queue connection URI.
  uri: ""
  # Dangerously flush all data. 
  dangerously_flush: false
  # represents the number of tasks a worker can handle simultaneously.
  num_workers: 2

ワーカープロセスの実行

バックグラウンドワーカーに選択した設定に応じて、2つの方法で実行できます。

Usage: demo_app start [OPTIONS]

Options:
  -w, --worker                     start worker
  -s, --server-and-worker          start same-process server and worker

実際のRedisキューを設定し、バックグラウンドジョブのみを実行するプロセスが必要な場合は、--workerを選択します。サーバーごとに1つのプロセスを使用できます。この場合、メインのWebまたはAPIサーバーはcargo loco startのみで実行できます。

$ cargo loco start --worker # starts a standalone worker job executing process
$ cargo loco start # starts a standalone API service or Web server, no workers

asyncバックグラウンドワーカーを設定した場合、ジョブは現在実行中のサーバープロセスの一部として実行されます。

例えば、--server-and-workerを実行する場合：

$ cargo loco start --server-and-worker # both API service and workers will execute

コードでバックグラウンドジョブを作成する

ワーカーを使用するには、主にジョブをキューに追加することを考えます。そのため、ワーカーをuseして後で実行します。

    // .. in your controller ..
    DownloadWorker::perform_later(
        &ctx,
        DownloadWorkerArgs {
            user_guid: "foo".to_string(),
        },
    )
    .await

RailsやRubyとは異なり、Rustではキューにシリアライズされてプッシュされる厳密に型付けされたジョブ引数を利用できます。

ワーカーから共有状態を使用する

グローバル状態を持つ方法を参照してください。一般的には、lazy_staticのようなものを使用して単一の共有状態を使用し、ワーカーからそれを参照するだけです。

この状態がシリアライズ可能であれば、WorkerArgsを介して渡すことを強く推奨します。

新しいワーカーの作成

ワーカーを追加するということは、引数を受け取り、ジョブを実行するバックグラウンドジョブロジックをコーディングすることです。また、locoにそれを知らせ、グローバルジョブプロセッサーに登録する必要があります。

workers/にワーカーを追加：

#[async_trait]
impl BackgroundWorker<DownloadWorkerArgs> for DownloadWorker {
    fn build(ctx: &AppContext) -> Self {
        Self { ctx: ctx.clone() }
    }
    async fn perform(&self, args: DownloadWorkerArgs) -> Result<()> {
        println!("[REDACTED_AWS_SECRET]========");
        println!("Sending payment report to user {}", args.user_guid);

        // TODO: Some actual work goes here...

        println!("[REDACTED_AWS_SECRET]========");
        Ok(())
    }
}

そしてapp.rsに登録します：

#[async_trait]
impl Hooks for App {
//..
    async fn connect_workers(ctx: &AppContext, queue: &Queue) -> Result<()> {
        queue.register(DownloadWorker::build(ctx)).await?;
        Ok(())
    }
// ..
}

ワーカーの生成

loco generateを使用してワーカーを自動的に追加するには、以下のコマンドを実行します。

cargo loco generate worker report_worker

ワーカージェネレーターは、アプリに関連付けられたワーカーファイルを作成し、ワーカーを検証するためのテストテンプレートファイルを生成します。

ワーカーの設定

config/<environment>.yamlでワーカーモードを指定できます。BackgroundAsyncとBackgroundQueueはジョブを非ブロッキングで処理し、ForegroundBlockingはジョブをブロッキングで処理します。

BackgroundAsyncとBackgroundQueueの主な違いは、後者がRedis/Postgres/SQLiteを使用してジョブを保存するのに対し、前者はRedis/Postgres/SQLiteを必要とせず、同じプロセス内で非同期的にメモリを使用することです。

# Worker Configuration
workers:
  # specifies the worker mode. Options:
  #   - BackgroundQueue - Workers operate asynchronously in the background, processing queued.
  #   - ForegroundBlocking - Workers operate in the foreground and block until tasks are completed.
  #   - BackgroundAsync - Workers operate asynchronously in the background, processing tasks with async capabilities.
  mode: BackgroundQueue

UIからのワーカー管理

Loco管理者ジョブプロジェクトでジョブキューを管理できます。

CLIによるジョブキューの管理

ジョブキュー管理機能は、アプリケーション内のジョブのライフサイクルを処理するための強力で柔軟な方法を提供します。これにより、ジョブのキャンセル、クリーンアップ、古いジョブの削除、ジョブ詳細のエクスポート、ジョブのインポートが可能になり、効率的で組織化されたジョブ処理が保証されます。

機能概要

  • ジョブのキャンセル
    指定した名前のジョブをキャンセルし、ステータスをcancelledに更新する機能を提供します。これは、不要になった、関連性のない、またはバグが検出された場合に処理を停止したいジョブに役立ちます。
  • ジョブのクリーンアップ
    既に完了またはキャンセルされたジョブを削除できます。これにより、不要なエントリを排除して、クリーンで効率的なジョブキューを維持できます。
  • 古いジョブのパージ
    日数で測定されるジョブの経過期間に基づいてジョブを削除できます。これは、古くて関連性のないジョブを削除することで、効率的なジョブキューを維持するのに特に役立ちます。
    注：--dumpオプションを使用してジョブの詳細をファイルにエクスポートし、エクスポートされたファイル内のジョブパラメータを手動で変更してから、import機能を使用して更新されたジョブをシステムに再導入できます。
  • ジョブ詳細のエクスポート
    すべてのジョブの詳細を指定された場所にファイル形式でエクスポートすることをサポートします。この機能は、バックアップ、監査、またはさらなる分析に役立ちます。
  • ジョブのインポート
    外部ファイルからのジョブのインポートを容易にし、外部ジョブデータをアプリケーションのワークフローにシームレスに統合できるようにします。

ジョブ管理コマンドにアクセスするには、以下のCLI構造を使用します。

ジョブキューの管理

使用法: demo_app-cli jobs [OPTIONS] <COMMAND>

コマンド:
  cancel 指定した名前のジョブをキャンセルし、ステータスを`cancelled`に設定します
  tidy 完了またはキャンセルされたジョブを削除します
  purge 日数に基づいてジョブを削除します
  dump すべてのジョブの詳細を指定されたフォルダー内のファイルに保存します
  import ファイルからジョブをインポートします
  help このメッセージまたは指定されたサブコマンドのヘルプを表示します

オプション:
  -e, --environment <ENVIRONMENT>  環境を指定します [デフォルト: development]
  -h, --help                       ヘルプを表示します
  -V, --version                    バージョンを表示します

ワーカーのテスト

Locoを使用してワーカーのバックグラウンドジョブを簡単にテストできます。ワーカーがForegroundBlockingモードに設定されていることを確認してください。これにより、ジョブがブロックされ、同期的に実行されます。ワーカーをテストする際、テストはワーカーが完了するまで待機するため、ワーカーが意図したタスクを達成したかどうかを検証できます。

すべてのワーカーテストを1か所にまとめるために、tests/workersディレクトリにテストを実装することをお勧めします。

さらに、ワーカージェネレーターを活用することもできます。これにより、テストが自動的に作成され、ライブラリでテストを設定する時間を節約できます。

テストの構造の例を以下に示します。

use loco_rs::testing::prelude::*;

#[tokio::test]
#[serial]
async fn test_run_report_worker_worker() {
    // Set up the test environment
    let boot = boot_test::<App, Migrator>().await.unwrap();

    // Execute the worker in 'ForegroundBlocking' mode, preventing it from running asynchronously
    assert!(
        ReportWorkerWorker::perform_later(&boot.app_context, ReportWorkerWorkerArgs {})
            .await
            .is_ok()
    );

    // Include additional assert validations after the execution of the worker
}
