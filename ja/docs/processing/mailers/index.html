メーラー

メーラーは、既存のlocoバックグラウンドワーカーインフラストラクチャを使用して、バックグラウンドでメールを配信します。すべてがシームレスに行われます。

メール送信

既存のメーラー（主にコントローラー内）を使用するには：

use crate::{
    mailers::auth::AuthMailer,
}

// in your controllers/auth.rs
async fn register(
    State(ctx): State<AppContext>,
    Json(params): Json<RegisterParams>,
) -> Result<Response> {
    // .. register a user ..
    AuthMailer::send_welcome(&ctx, &user.email).await.unwrap();
}

これにより、メール配信ジョブがキューに追加されます。配信は後でバックグラウンドで実行されるため、アクションは即座に完了します。

設定

メーラーの設定は、config/[stage].yamlファイルで行います。デフォルトの設定は次のとおりです。

# Mailer Configuration.
mailer:
  # SMTP mailer configuration.
  smtp:
    # Enable/Disable smtp mailer.
    enable: true
    # SMTP server host. e.x localhost, smtp.gmail.com
    host: {{ get_env(name="MAILER_HOST", default="localhost") }}
    # SMTP server port
    port: 1025
    # Use secure connection (SSL/TLS).
    secure: false
    # auth:
    #   user:
    #   password:

メーラーはSMTPサーバーにメールを送信することで機能します。SendGridを使用する場合の構成例（SMTPリレーオプションを選択）：

# Mailer Configuration.
mailer:
  # SMTP mailer configuration.
  smtp:
    # Enable/Disable smtp mailer.
    enable: true
    # SMTP server host. e.x localhost, smtp.gmail.com
    host: {{{ get_env(name="MAILER_HOST", default="smtp.sendgrid.net") }}
    # SMTP server port
    port: 587
    # Use secure connection (SSL/TLS).
    secure: true
    auth:
      user: "apikey"
      password: "your-sendgrid-api-key"

デフォルトのメールアドレス

すべてのメール送信タスクでメールアドレスを指定する以外に、メーラーごとにデフォルトのメールアドレスをオーバーライドできます。

まず、Mailerトレイトのopts関数をオーバーライドします。この例ではAuthMailerの場合です：

impl Mailer for AuthMailer {
    fn opts() -> MailerOpts {
        MailerOpts {
            from: // set your from email,
            ..Default::default()
        }
    }
}

開発環境でのメールキャッチャーの使用

MailHogやmailtutan（Rustで記述）のようなアプリを使用できます。

$ cargo install mailtutan
$ mailtutan
listening on smtp://0.0.0.0:1025
listening on http://0.0.0.0:1080

これにより、ローカルのSMTPサーバーが立ち上がり、http://localhost:1080で、受信したメールを「キャッチ」して表示する便利なUIが利用できます。

次に、これをdevelopment.yamlに追加します。

# Mailer Configuration.
mailer:
  # SMTP mailer configuration.
  smtp:
    # Enable/Disable smtp mailer.
    enable: true
    # SMTP server host. e.x localhost, smtp.gmail.com
    host: localhost
    # SMTP server port
    port: 1025
    # Use secure connection (SSL/TLS).
    secure: false

これで、メーラーワーカーはlocalhostのSMTPサーバーにメールを送信します。

メーラーの追加

メーラーを生成できます。

cargo loco generate mailer <mailer name>

または、仕組みを理解したい場合は手動で定義することもできます。mailers/auth.rsに以下を追加します。

static welcome: Dir<'_> = include_dir!("src/mailers/auth/welcome");
impl AuthMailer {
    /// Sending welcome email the the given user
    ///
    /// # Errors
    ///
    /// When email sending is failed
    pub async fn send_welcome(ctx: &AppContext, _user_id: &str) -> Result<()> {
        Self::mail_template(
            ctx,
            &welcome,
            Args {
                to: "foo@example.com".to_string(),
                locals: json!({
                  "name": "joe"
                }),
                ..Default::default()
            },
        )
        .await?;
        Ok(())
    }
}

各メーラーには、意見が反映された、事前に定義されたフォルダー構造があります。

src/
  mailers/
    auth/
      welcome/      <-- メールとすべてのテンプレートのすべての部分
        subject.t
        html.t
        text.t
    auth.rs         <-- メーラー定義

メーラーの実行

メーラーはバックグラウンドワーカーとして動作します。つまり、ジョブを処理するにはワーカーを個別に実行する必要があります。デフォルトの起動コマンドcargo loco startはワーカーを起動しないため、個別に実行する必要があります。

ワーカーを実行するには、次のコマンドを使用します。

cargo loco start --worker

サーバーとワーカーを同時に実行するには、次のコマンドを使用します。

cargo loco start --server-and-worker

テスト

ワークフローの一部として送信されるメールのテストは複雑なタスクであり、ユーザー登録時のメール確認やユーザーパスワードメールのチェックなど、さまざまなシナリオの検証が必要です。主な目的は、ワークフローで送信されたメールの数を確認し、メールの内容をレビューし、データスナップショットを許可することで、テストプロセスを合理化することです。

Locoでは、スタブテストメール機能を導入しました。基本的に、メールは実際に送信されず、代わりにテストコンテキストの一部として、メールの数とその内容に関する情報を収集します。

設定

テストでスタブを有効にするには、YAMLファイルのメーラーセクションに次のフィールドを追加します。

mailer:
  stub: true

注意：メール送信者がワーカープロセス内で動作する場合、ワーカーモードがForegroundBlockingに設定されていることを確認してください。

スタブを設定したら、単体テストに進み、以下の例に従ってください。

テストの記述

テストの説明:

  • コードの一部としてメール送信を担当するエンドポイントへのHTTPリクエストを作成します。
  • コンテキストからメーラーインスタンスを取得し、送信されたメールの数と内容に関する情報を含むdeliveries()関数を呼び出します。
use loco_rs::testing::prelude::*;

#[tokio::test]
#[serial]
async fn can_register() {
    configure_insta!();

    request::<App, Migrator, _, _>(|request, ctx| async move {
        // Create a request for user registration.

        // Now you can call the context mailer and use the deliveries function.
        with_settings!({
            filters => cleanup_email()
        }, {
            assert_debug_snapshot!(ctx.mailer.unwrap().deliveries());
        });
    })
    .await;
}
